Project Overview

The Critical Code Reviewer (CCR) Framework is a responsible AI initiative designed to help educational technology developers identify potential fairness, accessibility, inclusion, pedagogy, transparency, and privacy concerns during software development and code review.

Traditional code-review tools often focus primarily on functionality, performance, security, and maintainability. CCR extends this process by evaluating code through a human-centered and ethical lens, helping developers consider how software decisions may affect diverse learner populations.

The project was developed within the Arizona State University Learning Engineering Institute to support the creation of more equitable, inclusive, transparent, and learner-centered educational technologies.

Problem Statement

Educational technologies increasingly influence learning experiences, recommendations, assessments, progression decisions, and access to educational opportunities. However, many software-development workflows do not systematically evaluate the ethical and social implications of code before deployment.

CCR addresses this gap by translating responsible AI and inclusive-design principles into practical dimensions, criteria, definitions, and reviewable indicators that developers can apply during code review.

Project Goals

  • Develop an evidence-informed framework for ethical and responsible code review.
  • Translate responsible AI principles into practical guidance for software developers.
  • Support inclusive and equitable educational technology development.
  • Help developers recognize fairness, accessibility, transparency, privacy, and learner-impact risks in code.
  • Support the development of an AI-assisted code-review tool for educational technology contexts.
  • Validate and refine the framework using feedback from educational technology stakeholders and software developers.

Framework Dimensions

Fairness Evaluation

Examines whether training and evaluation data adequately represent the learner populations and educational settings the system is intended to serve. It also considers whether group-level fairness targets and evaluation procedures are defined.

Pedagogy

Evaluates whether adaptive educational systems support meaningful learner progression, transparent learning requirements, appropriate early-warning decisions, and pedagogically grounded recommendations.

Decision Fairness

Assesses how protected or proxy attributes are used in decision logic and whether fairness protections are clearly defined, validated, enforced, and separated from inappropriate decision inputs.

Inclusion

Identifies culturally narrow assumptions, exclusionary language, inaccessible user flows, and design decisions that may disadvantage learners with different identities, abilities, devices, languages, locations, or levels of connectivity.

Transparency

Evaluates whether AI-driven outputs are understandable, explainable, appropriately qualified, and supported by human review, contestability, and override mechanisms.

Privacy

Assesses whether learner data are collected, processed, stored, and shared with appropriate consent checks and safeguards.

Example Evaluation Criteria

  • Unrepresentative Dataset Coverage
  • Undefined Fairness Evaluation Attributes
  • Missing Group-Wise Fairness Targets
  • Unvalidated Early-Warning Thresholds
  • Persistent Negative State
  • Hidden Learning Path Requirements
  • Protected and Proxy-Based Differential Treatment
  • Unvalidated Fairness Mitigations
  • Culturally Narrow Representations and Validation
  • Exclusionary Language in Code and UI
  • Excluding Access Assumptions
  • Opaque AI Decision Outputs
  • Causal Overclaiming in Analytics
  • Automation Without Review Safeguards
  • Missing Consent Checks for Learner Data

Summer 2026 Validation Study

Study Purpose

The purpose of the validation study is to evaluate and refine the CCR Framework using feedback from educational technology stakeholders and software developers.

The study will first identify ethical, accessibility, inclusion, equity, transparency, privacy, accountability, and trust concerns experienced by people who use educational software. The framework will then be refined and evaluated by software developers during code review.

Primary Research Question

How well does the Critical Code Reviewer Framework address the ethical, inclusion, accessibility, equity, transparency, and trust concerns experienced by educational technology stakeholders, and how do software developers perceive its clarity, usefulness, completeness, and practicality during code review?

Research Design

The study uses a sequential, qualitative-dominant mixed-methods and user-centered framework-development design. Qualitative evidence will serve as the primary basis for framework refinement, while quantitative survey findings will complement the qualitative results.

Findings from the first phase will be used to revise the initial framework and produce Version 2.0. Software developers will then evaluate Version 2.0 during the second phase. Integrated findings will guide the development of Version 3.0.

Validation Phase 1: Educational Technology Stakeholder Evaluation

Approximately 10–15 educational technology stakeholders will participate in individual research sessions. Participants may include undergraduate students, graduate students, instructors, and other people with experience using educational software.

Activities

  • Brief demographic questionnaire
  • Semi-structured interview about experiences using educational software
  • Discussion of fairness, accessibility, inclusion, transparency, privacy, accountability, trust, and responsible AI concerns
  • Researcher field notes and audio recording with participant consent

The CCR Framework will not be introduced before the interview. This will allow participants to describe concerns based on their own experiences rather than being influenced by the existing framework.

Findings will be analyzed using reflexive thematic analysis and compared with the existing framework. The results will help identify concerns that are already represented, missing, unclear, or insufficiently addressed.

Findings from this phase will inform the development of Version 2.0 of the CCR Framework.

Validation Phase 2: Software Developer Evaluation

Approximately 5–8 software developers or computer science participants with software-development experience will evaluate Version 2.0 of the CCR Framework.

Activity 1: Hybrid Card Sorting

Participants will use FigJam to organize CCR review criteria under the framework dimensions they believe best represent them.

Participants may move criteria between dimensions, identify overlapping or redundant dimensions, create new categories, and explain the reasoning behind their decisions. This activity will evaluate the framework's conceptual organization, clarity, and completeness.

Activity 2: Think-Aloud Usability Testing

Participants will complete a code-review task using the CCR Tool, implemented as a GitHub Action. They will verbalize their thoughts, interpretations, decisions, and reactions while applying the framework during code review.

The session will examine how developers interpret the framework, how they use its dimensions and criteria, where they experience confusion, which ethical concerns the tool helps identify, and what improvements they recommend.

Activity 3: Post-Task Evaluation Survey

Participants will complete the System Usability Scale to evaluate the usability of the CCR Tool. They will also answer framework-specific questions about:

  • Clarity
  • Conceptual organization
  • Usefulness
  • Completeness
  • Practicality

Research Methods

  • Semi-structured interviews
  • Demographic questionnaires
  • Hybrid card sorting using FigJam
  • Think-aloud usability testing
  • System Usability Scale
  • Framework-specific evaluation survey
  • Observational field notes
  • Reflexive thematic analysis
  • Descriptive statistical analysis

Data Analysis Plan

Interview and think-aloud transcripts will be analyzed using reflexive thematic analysis. The analysis will identify recurring concerns, usability challenges, unclear framework terminology, missing criteria, strengths, and recommendations for improvement.

Card-sorting results will be examined for agreement in how criteria are grouped, criteria that are repeatedly reassigned, dimensions perceived as overlapping, unclear terminology, and newly proposed categories.

System Usability Scale responses will be converted into scores ranging from 0 to 100. Descriptive statistics will be used to summarize perceived usability. Framework-specific survey responses will also be summarized using frequencies, percentages, means, and standard deviations.
