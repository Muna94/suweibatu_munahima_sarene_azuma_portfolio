Project Overview

The Critical Code Reviewer (CCR) Framework is a responsible AI initiative designed to help educational technology developers identify potential fairness, pedagogy, decision fairness, inclusion, transparency, privacy, and AI governance concerns during software development and code review.

Traditional code-review tools often focus primarily on functionality, performance, security, and maintainability. CCR extends this process by evaluating code through a human-centered and ethical lens, helping developers consider how software decisions may affect diverse learner populations.

The project was developed within the Arizona State University Learning Engineering Institute to support the creation of more equitable, inclusive, transparent, accountable, and learner-centered educational technologies.

Problem Statement

Educational technologies increasingly influence learning experiences, recommendations, assessments, progression decisions, student-support interventions, and access to educational opportunities. However, many software-development workflows do not systematically evaluate the ethical, educational, and social implications of code before deployment.

CCR addresses this gap by translating responsible AI, inclusive-design, accessibility, and educational principles into practical dimensions, criteria, definitions, and reviewable indicators that developers can apply during code review.

Project Goals

  • Develop an evidence-informed framework for ethical and responsible code review.
  • Translate responsible AI principles into practical guidance for software developers.
  • Support inclusive and equitable educational technology development.
  • Help developers identify fairness, pedagogy, accessibility, transparency, privacy, learner-impact, and AI governance risks in code.
  • Support the development of an AI-assisted code-review tool for educational technology contexts.
  • Validate and refine the framework using feedback from educational technology stakeholders and software developers.

Framework Dimensions and Evaluation Criteria

The CCR Framework currently includes seven dimensions containing both code-level and process-level evaluation criteria.

Fairness Evaluation

Fair & Representative Training Data Assurance

Addresses problems in data selection, fairness-group definitions, group-level targets, and the thresholds used to evaluate fairness. This dimension focuses on how fairness is measured before examining later decision behavior.

Evaluation Criteria

  • Unrepresentative Dataset Coverage
  • Undefined Fairness Evaluation Attributes
  • Missing Group-Wise Fairness Targets

Pedagogy

Adaptive Fairness & Transparent Learner Progression Integrity

Addresses situations where automation, predictions, and adaptive decisions do not align with educational purpose or remove necessary human judgment. This dimension focuses on learning quality, learner progression, and appropriate educational use.

Evaluation Criteria

  • Unvalidated Early-Warning Thresholds
  • Persistent Negative State
  • Hidden Learning Path Requirements
  • Pedagogically Unconstrained Adaptation

Decision Fairness

Protected Attribute Governance & Validated Fairness Enforcement

Addresses problems in decision rules, model behavior, proxy-variable use, protected-attribute handling, and fairness controls that influence how learners are treated. This dimension focuses on the decision path itself.

Evaluation Criteria

  • Protected and Proxy-Based Differential Treatment
  • Unvalidated Fairness Mitigations
  • Unenforced Protected-Attribute Separation

Inclusion

Cultural & Accessibility Equity in Design and Code

Addresses design and code decisions that may exclude learners through interface language, validation rules, navigation, defaults, disability access barriers, device requirements, connectivity assumptions, or culturally narrow representations.

Evaluation Criteria

  • Culturally Narrow Representations and Validation
  • Exclusionary Language in Code and UI
  • Excluding Access Assumptions

Transparency

Transparent, Explainable, Validated & Human-Supervised AI Decisioning

Addresses problems in explanations, labels, analytics language, prediction use, and result framing that make system behavior difficult to understand or easy to misinterpret. It also examines whether people can review, challenge, override, or provide context for consequential algorithmic decisions.

Evaluation Criteria

  • Opaque AI Decision Outputs
  • Causal Overclaiming in Analytics
  • Mismatched Prediction Validation and Use
  • Unidirectional Algorithmic Decisions Without Educator Contestability
  • Automation Without Review Safeguards

Privacy

Addresses problems related to the consent, access, use, retention, sharing, and protection of learner information. This dimension focuses on the safe and respectful handling of personal and educational data.

Evaluation Criterion

  • Missing Consent Checks for Learner Data

AI Governance

Addresses problems in review gates, release control, monitoring, accountability, fairness validation, and ongoing checks for AI-related software changes. This dimension focuses on managing AI risk throughout development, deployment, and continued use.

Evaluation Criteria

  • Unenforced High-Risk AI Release Gates
  • Missing Post-Deployment Risk Monitoring
  • Missing Fairness Review and Validation for Student-Affecting Changes
  • Missing Bias Review for AI-Assisted Code
  • Missing Bias Regression Checks for AI-Generated Code
  • Missing User-Need Traceability
  • Untracked Fairness Mitigation Outcomes

Framework Validation Study — Summer 2026–Present

Following the development of the initial Critical Code Reviewer Framework, I am currently contributing to a user-centered validation study designed to evaluate and refine the framework through feedback from educational technology stakeholders and software developers.

The study investigates whether the framework adequately addresses responsible AI concerns, including fairness, accessibility, inclusion, transparency, privacy, accountability, trust, pedagogy, and AI governance.

It also evaluates the framework’s clarity, usefulness, completeness, conceptual organization, and practicality during software code review.

Research Design

The validation study uses a sequential, qualitative-dominant mixed-methods design consisting of two phases.

Phase 1: Educational Technology Stakeholders

  • Demographic questionnaire
  • Semi-structured interviews
  • Identification of stakeholder concerns
  • Reflexive thematic analysis
  • Refinement of the CCR Framework into Version 2.0

Phase 2: Software Developer Evaluation

  • Developer demographic questionnaire
  • Hybrid card sorting using FigJam
  • Think-aloud usability testing
  • System Usability Scale
  • Framework-specific evaluation survey
  • Follow-up semi-structured interviews
  • Development of Version 3.0 of the CCR Framework

Current Status

The validation study is currently in development. Study materials, participant procedures, research instruments, and data-analysis plans are being finalized in preparation for participant recruitment and data collection.

Expected Outcome

The study is expected to provide evidence about how well the CCR Framework represents stakeholder concerns and whether software developers find it clear, useful, complete, practical, and usable during code review.

Integrated findings from both validation phases will guide refinement of the framework and tool and support the development of CCR Version 3.0.
